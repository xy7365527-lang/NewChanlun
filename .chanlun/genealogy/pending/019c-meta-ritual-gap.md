# 019c — 元编排进化无仪式门控（自指性缺口）

**状态**: 生成态
**类型**: meta-rule
**日期**: 2026-02-18
**溯源**: [新缠论]
**域**: 元编排方法论（仪式架构）

---

## 矛盾描述

元编排体系有两个层次，门控机制不对称：

| 层次 | 变更路径 | 门控 |
|------|---------|------|
| 域层（缠论定义） | 分离 → pending/ → 实证 → /ritual → settled/ | **有仪式** |
| 元层（SKILL.md/CLAUDE.md/agent定义） | meta-observer发现 → 直接修改文件 | **无仪式** |

这是自指性矛盾：元编排教导域层"结算是仪式"（SKILL.md），但元编排自身的变更不经过仪式。

## 发现过程

### 故障场景

meta-observer 识别到"暗规则" → 编排者口头认可 → agent 直接修改 SKILL.md → 无谱系记录、无回溯扫描、无基底刷新 → 质询序列退化 → 概念层错误未被捕获。

### 关键观察

1. `/ritual` 定义的写入目标只有域层文件（definitions.yaml, docs/spec/, 缠论知识库.md），不包含 SKILL.md/CLAUDE.md/agent定义
2. meta-observer.md 第82行写着"不直接修改 SKILL.md 或 CLAUDE.md（仪式的事）"——承认需要仪式但没有路径
3. 四分法（018）已提供判据：定理和行动类不需要仪式，选择和语法记录类需要仪式
4. 这个判据在域层和元层应当统一，但当前只在域层被执行

### 自指性分析

"给元编排加仪式门控"本身就是一次元编排变更。回退链终止于编排者——编排者授权是不需要仪式的（005b：系统的语法规则不能自我奠基）。015 已承认这一点："本文档处于生成态"就是承认自我奠基不可能。

## 修复方向

**扩展 /ritual 覆盖元层，而非创建独立 /meta-ritual。** 理由：
- 域层和元层的仪式四步骤完全相同（写入定义、写入谱系、回溯扫描、基底刷新）
- 两套仪式违反 DRY，增加认知负担，按016更容易被跳过

**四分法过滤器**：
- 定理类元变更 → 不走仪式，但必须写谱系
- 选择类元变更 → 走 /ritual
- 语法记录类元变更 → 走 /ritual
- 行动类元变更 → 不走仪式，不写谱系

**元编排版本化**：不引入语义化版本号（git 已有完整历史），在 YAML 头部增加 `last_ritual: [谱系ID]` 字段。

## 推导链

1. 域层有仪式门控（/ritual），元层没有（事实）
2. meta-observer 定义承认"不该直接改"但没提供替代路径（事实）
3. 018 四分法已提供"哪些变更需要仪式"的判据（已结算）
4. 015 证明方法否定自身形式是结构必然（已结算）
5. ∴ 元层需要与域层相同的仪式门控，且仪式定义本身处于生成态
