# 132号：historical_tension 边类型缺失——活跃张力与历史失效张力的 Schema 区分

**类型**: 矛盾发现（方法论批评层面）
**状态**: 生成态
**日期**: 2026-02-22
**来源**: 三轮 Gemini 双向严格讨论——焦点5第三轮 Gemini 矛盾点5

## 发现

### Gemini 的批评

Gemini 第三轮矛盾点5指出：

> "如果系统依赖外部的文本节点（121号）来解释某条结构边（tensions_with）'其实已经失效'，说明代码实现（图谱Schema）没有忠实反映定义。必须引入新的边类型（如 `historical_tension` 或 `negated_tension`），否则图谱的自动化推理引擎将无法区分'活跃张力'和'历史遗留'，导致推理污染。"

### 当前系统状态

以 T16（062 ↔ 065）为典型案例：

- **结构边**：`062 tensions_with 065`（在 dag.yaml 中存在，类型为 tensions_with）
- **121号结论**：该边"在068否定065后自动失效"，但"作为历时性记录保留（不删除）"
- **120号结论**：062被扬弃后（negated_by: [064, 093]），其引用自动降级为"历史参照"

**问题**：dag.yaml 中这条边的类型依然是 `tensions_with`，而 `tensions_with` 的语义在系统中是"当前活跃的未解决张力"。但121号说它已失效，两者语义冲突——图谱结构与谱系文本解释不一致。

### 问题的准确范围

这是一个**Schema 设计与语义一致性**问题，不是"矛盾降级"问题：

- 系统没有"将 negates 软化为 tensions_with"（矛盾降级）
- 系统有"将历史失效的 tensions_with 和当前活跃的 tensions_with 混用同一边类型"（语义混用）

两个问题性质不同：
- 矛盾降级：谴责系统在**分类**上作假（本该是negates却标成tensions_with）
- 语义混用：谴责系统在**时态**上不精确（失效的和活跃的用同一标签）

### 语义混用的范围

受影响的 tensions_with 边包括：
- **T16**（062 ↔ 065）：062已被否定（negated_by），065已被068否定——两端节点均已失效或被取代，张力应为"历史失效"
- 其他涉及已失效节点（negated_by 非空）的 tensions_with 边（需专项扫描）

## 三种处理路径

### 路径A：引入 historical_tension 边类型
在 dag.yaml Schema 中增加 `historical_tension` 边类型，将所有"历时性记录"的失效张力边改为此类型。
- 优点：图谱语义精确，自动化推理引擎可区分
- 成本：需要重新分类现有 tensions_with 边，边界判断需要一套规则

### 路径B：引入 valid_until 字段
保留 tensions_with 边类型，但增加 `valid_until: <节点id>` 字段，表示"在该节点的否定事件发生后失效"。
- 优点：不改变现有边类型分类
- 成本：查询时需要额外检查 valid_until 字段

### 路径C：维持现状 + 完善文本谱系
接受"图谱结构 + 谱系文本"的双层表达——图谱记录历时状态，谱系文本解释当前有效性。
- 优点：无需改动 dag.yaml Schema
- 缺点：自动化推理依赖文本谱系，非严格

## 分类

本条属于"选择"（三种路径，需价值判断），但优先级低于131号（127号下游补充）。

当前系统未实现自动化推理引擎（图谱主要供人工读取），因此路径C在短期内是可接受的，但路径A或B应作为图谱工程化的长期目标。

## 边界条件

- 翻转条件：如果实现了基于 dag.yaml 的自动化推理引擎，路径C不可接受，必须走路径A或B
- 如果存在更多历史失效的 tensions_with 边（扫描结果），路径A的成本收益比提高
