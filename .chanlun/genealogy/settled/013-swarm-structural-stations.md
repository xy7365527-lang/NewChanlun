## 矛盾描述

2026-02-17 ceremony 中，蜂群多次出现"不按元编排全部流程走"的偏差：

- 三个搜索工位完成后，直接汇总给编排者，跳过谱系写入
- 谱系写入后，方法论洞察未回写元编排系统
- Serena 未在 ceremony 启动时加载，导致跨会话规则丢失

**根因**：当前蜂群模型是纯任务驱动——每次只按任务拆工位。谱系写入、元编排进化、Serena 同步等**不以任务为转移的常设功能**没有被编入 agent team 设计，所以每次拉蜂群时自动丢失。

## 发现过程

编排者指出："agent teams 有时候不按照元编排的全部流程走"，进而辨认出：有些工位不以任务为转移，它们是蜂群的**结构性基础设施**，不是任务的派生物。

## 结论（语法记录）

**蜂群的工位分为两类：结构工位（常设）和任务工位（临时）。**

### 结构工位（Structural Stations）

| 工位 | 触发条件 | 职责 | 依据 |
|------|---------|------|------|
| 谱系工位 | 每次概念发现后 | 立即写入谱系，检查张力 | 012号谱系 |
| 元编排进化工位 | 每次方法论洞察后 | 回写 SKILL.md + Serena 记忆 | 编排者指示 |
| Serena 同步工位 | 每次 commit 时 | 同步跨会话状态到 Serena 记忆 | 编排者指示 |
| 质量守卫工位 | 每次产出前 | 检查结果包六要素、谱系引用 | CLAUDE.md 原则#3 |

### 任务工位（Task Stations）

随任务创建和销毁。例如：原文搜索、代码实现、数据拉取、测试执行。

### 架构规则

```
蜂群实例化时：
  1. 必须先挂载全部结构工位（常设）
  2. 再按任务拆分任务工位（临时）
  3. 结构工位不随任务完成而销毁
  4. 结构工位通过事件触发，不通过任务分配触发

蜂群循环：
  任务工位A完成 → [谱系工位] 写入 → [质量守卫] 检查
  任务工位B完成 → [谱系工位] 写入 → [质量守卫] 检查
  方法论洞察产生 → [元编排进化工位] 回写
  commit 执行     → [Serena同步工位] 同步
```

### 类比

- **操作系统**：内核服务（常驻）vs 用户进程（按需）
- **010号谱系的映射**：结构工位 = 构造层（始终运行的递归基础设施），任务工位 = 分类层（随具体走势生灭的实体）

## 对 agent team 实现的影响

1. `SKILL.md` 的蜂群初始化流程必须包含"挂载结构工位"步骤
2. 每个 agent team prompt 必须包含结构工位的职责描述
3. agent team 的完成条件不仅是"任务完成"，还包括"结构工位确认无遗漏"
