# 028 — ceremony 无工位时的默认动作

**类型**: 语法记录
**状态**: 已结算
**日期**: 2026-02-19
**前置**: 027-positive-instruction-over-prohibition

## 矛盾现象

ceremony 热启动完成后，若蜂群评估为"无活跃工位"，系统输出状态快照并停止。这与 027号原则（正面指令优于禁止性规则）矛盾——"无工位"不应该是停止信号，应该是扫描信号。

## 推导链

1. 027号：正面指令 > 禁止性规则 → 每个决策点需要默认动作
2. post-commit-flow 已解决 commit 后的停顿：`→ 接下来：[动作]`
3. ceremony 完成是另一个决策点，但缺少默认动作
4. "无活跃工位"≠"无可做之事"——它只意味着没有显式追踪的工位
5. 默认动作应为：扫描代码库（TODO/覆盖率/spec合规/谱系张力），产出至少一个工位

## 已结算原则

**ceremony 完成后，无论是否有活跃工位，下一个输出必须是行动声明 + tool 调用。**

- 有工位：`→ 接下来：[恢复最高优先级工位]`
- 无工位：`→ 接下来：扫描可推进工作`，然后执行扫描（TODO / 覆盖率 / spec 合规 / 谱系张力）

不允许以"等待外部输入"结束 ceremony。如果扫描后确实无事可做，输出扫描结果作为状态快照（这本身就是行动的产出，不是等待）。

## 谱系链接

- **027**: 正面指令优于禁止性规则（本条的前置原则）
- **020**: ceremony 本体论定位（ceremony 是相位转换节点）
- **post-commit-flow**: 同一病灶的 commit 端修复

## 影响

- ceremony skill 需要更新：蜂群评估部分的输出格式
- 所有"无工位"路径都需要默认扫描动作
