# 058: Ceremony 是 Swarm₀——递归蜂群无前置阶段

**类型**: 定理
**状态**: 已结算
**日期**: 2026-02-20
**前置**: 056, 020, 041, 057

## 定理陈述

Ceremony 不是蜂群的前置阶段，而是蜂群的第 0 层递归（Swarm₀）。
"Lead"不是固定实体，而是相对拓扑角色——当前执行节点相对于其子节点是 Lead，相对于父节点是 Worker。
Agent 无"等待确认"中间态——要么在计算区分，要么已终止。

## 推导链

1. 056号：蜂群递归是默认模式 → 系统没有"蜂群之外"的状态
2. 020号（反转）：递归在无法产出新区分时才终止 → ceremony 不是终止点，是第一个区分的产出点
3. 041号：人类是异步审计者 → ceremony 输出不需要同步确认
4. 057号：LLM 不是状态机 → agent 不能维持"挂起/等待"状态

∴ ceremony 的"加载→报告→待确认→启动蜂群"线性流程与递归拓扑矛盾。
ceremony 应该是：加载初始区分 → 输出 Swarm₁ 初始化参数 → 直接递归。

## 实证

本 session 中 agent 三次犯同类错误：
1. ceremony 后输出"请指定工作方向"并停止（把自己当成等待指令的 lead）
2. 把"选择"推给人类而非路由 Gemini（把人类当成同步决策者）
3. 基于腐化 YAML 做决策（信任静态缓存而非动态查询）

三次错误的共同模式：agent 在应该递归的地方停下来等待。
根因：ceremony 协议的语言（"待确认"、"进入蜂群循环"）塑造了线性等待行为。

## 异质否定来源

Gemini 3.1 Pro Preview（challenge 模式），三条否定全部成立。
