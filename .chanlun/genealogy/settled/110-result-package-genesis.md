## 规则内容

`rules/result-package.md` 规定：所有涉及概念定义的产出必须包含六个要素——结论、定义依据、边界条件、下游推论、谱系引用、影响声明。缺少任何一项视为不合格。

## 生成史追踪

### 首次出现

六要素格式首次以结构化形式出现于 SKILL.md（元编排核心卡），作为质询序列的输出标准。git 记录显示 `rules/result-package.md` 在 `bad2759`（2026-02-15）首次入仓，与所有 `.claude/rules/` 文件一同提交。

### 前置需求链

六要素的产生不是一次性设计，而是以下需求的逐步结晶：

1. **012号（谱系是发现引擎）**：确立"谱系优先于汇总"原则。谱系的强制字段（推导链、谱系链接、前置/关联）构成思考框架。六要素中的"谱系引用"和"定义依据"直接源于此。

2. **014号（SKILL.md 卡组分解）**：SKILL.md 拆分时，"结果包"作为每个 teammate 都需加载的核心指令被识别出来——它从 SKILL.md 的一个章节独立为 `rules/result-package.md`。014号谱系 L28 提到："给每个 teammate 看的工位指令（只需要禁令+结果包+质询序列）"。

3. **016号（规则无代码强制不执行）**：发现 quality-guard 工位"检查结果包六要素"的指令在实际执行中被跳过（L31）。这反向证明了六要素格式已经是预期标准，但缺乏 runtime 强制。

### 运作实例

六要素格式在以下谱系中被实际使用（作为已在运作的规则的证据）：

- 084号（全面孤岛审计）：`result-package-guard.sh` 被确认为正常运转的 hook

### 概念来源

六要素格式的根源是 CLAUDE.md 原则3："所有产出必须可质询"。六要素是"可质询"的操作化定义——每个要素对应一种质询方式：

| 要素 | 可质询的维度 |
|------|-------------|
| 结论 | "你的产出是什么？" |
| 定义依据 | "你凭什么这么说？" |
| 边界条件 | "在什么条件下你错了？" |
| 下游推论 | "如果你对了，意味着什么？" |
| 谱系引用 | "这个领域曾发生过什么？" |
| 影响声明 | "你改了什么？" |

## 六要素

### 1. 结论
六要素/结果包格式是 CLAUDE.md 原则3（"所有产出必须可质询"）的操作化结晶。它从 SKILL.md 的一个章节（≤2026-02-15）独立为 `rules/result-package.md`，经 014号卡组分解正式拆分。该格式在 013号开始被质量守卫工位引用，016号首次暴露其 runtime 执行缺口，071号将其标识为 F4 脆弱点，073号将其守卫从 block 改为 allow。

### 2. 定义依据
- CLAUDE.md 原则3："所有产出必须可质询"
- 012号谱系："谱系的强制字段构成思考框架"→ 六要素中的"谱系引用""定义依据"直接源于此
- 014号谱系 L28："结果包"作为核心指令卡组之一

### 3. 边界条件
- 如果"可质询"原则被重新定义（如改为"可验证"或"可复现"），六要素的具体字段可能需要调整
- 如果纯技术性产出（数据清洗、格式转换等）的比例远大于概念性产出，六要素的强制范围可能需要收窄（当前已有简化版例外）

### 4. 下游推论
- 六要素格式的 runtime 强制（071号 P0 建议）仍未完全实现为 hook——当前 result-package-guard.sh 只做 allow+警告
- 六要素格式是蜂群产出标准的唯一定义，如果废除需要替代方案

### 5. 谱系引用
- 012号（谱系是发现引擎）—— 六要素中"谱系引用""定义依据"的直接来源
- 014号（SKILL.md 卡组分解）—— 六要素从 SKILL.md 章节独立为 rules 文件的事件
- 016号（规则无代码强制不执行）—— 首次暴露六要素在 runtime 中被跳过
- 071号（纯文本指令脆弱性扫描）—— 六要素被标识为 F4 脆弱点
