# 111 — 不允许绕过概念矛盾禁令的生成史

## 规则内容

`rules/no-workaround.md` 规定：当遇到定义之间的冲突、逻辑上不可弥合的矛盾、规格与实现之间的根本不一致时，禁止写 workaround 让代码先跑通，必须停下来精确描述矛盾并使用 `/escalate` 上浮。

## 生成史追踪

### 首次出现

`rules/no-workaround.md` 在 `bad2759`（2026-02-15）首次入仓，与所有 `.claude/rules/` 文件一同提交。但其内容表明该禁令早已在编排者的实践中运作——文件只是将已有实践写成显式规则。

### 前置概念链

no-workaround 禁令的产生是以下概念的逻辑必然推论：

1. **005b号（对象否定对象语法规则）**：确立"对象否定对象"为走势描述语言的语法规则。no-workaround 是该规则在元编排层的镜像——概念矛盾是对象（定义之间的冲突），workaround 是"非对象来源的否定"（用代码补丁掩盖定义冲突），因此 workaround 在蜂群中是语法不合法的。

2. **012号（谱系是发现引擎）**："矛盾不是bug。矛盾是这个系统最有价值的产出。" 这句话是 no-workaround 禁令的精神来源。如果矛盾是最有价值的产出，那么绕过矛盾 = 丢弃最有价值的产出。

3. **CLAUDE.md 原则1（概念优先于代码）**："定义不清楚时不写代码。" no-workaround 是原则1的反面表述——原则1说"正面该做什么"（先搞清定义），no-workaround 说"反面不许做什么"（不许绕过定义冲突）。

### 运作实例

no-workaround 禁令在以下事件中被实际触发或引用：

- **050号（守恒约束语义空洞）**：发现"方向守恒 ≠ 资本守恒"是概念矛盾而非实现错误。系统停下来记录矛盾而非 workaround。
- **071号（纯文本指令脆弱性扫描）**：no-workaround 被标识为 F2 脆弱点（#12），因为"必须做的5步"（停下来、描述矛盾、/escalate）缺乏 runtime 强制。
- **083号（干净终止过早诊断）**：workaround 被明确拒绝为策略选项。
- **090号（严格性语法规则）**：no-workaround 与 no-patch-mentality 被确认为互补禁令——前者禁止绕过矛盾（该停不停），后者禁止非严格的产出（该严格不严格）。

### 与 090号的关系

090号确立"严格性是蜂群的语法规则"后，no-workaround 的存在论位置被进一步明确：

- no-workaround（CLAUDE.md 原则2）= 禁止绕过矛盾（否定面）
- no-patch-mentality（原则17 的操作化）= 要求所有产出严格（肯定面）
- 两者是同一规则的正反两面：严格的产出不会绕过矛盾，不绕过矛盾的产出趋向严格

### 编排者意图的追溯

no-workaround 禁令的产生事件无法精确定位到单一 commit（它在 bad2759 随全部 rules 文件一起入仓），但其内容的"语气"——"此规则优先级高于所有其他规则"——表明编排者在此之前已多次遇到 LLM agent 绕过概念矛盾的行为，并将"停下来"确立为核心纪律。

这与 005b 号的 Schelling 式命名具有结构同构：005b 是给走势描述语言命名一条语法规则，no-workaround 是给蜂群元编排命名一条纪律。两者都是"将已在运作的规则从隐性变为显式"。

## 六要素

### 1. 结论
no-workaround 禁令是 005b号（对象否定对象）在元编排层的镜像，是 012号（矛盾是最有价值的产出）的操作化推论。它在 2026-02-15 随 `.claude/rules/` 入仓，但内容表明编排者的实践中该规则已先于文件存在。090号确立严格性语法规则后，no-workaround 的位置被确认为"严格性"的否定面。

### 2. 定义依据
- 005b号：对象否定对象语法规则 → workaround 是非对象否定（语法不合法）
- 012号："矛盾是这个系统最有价值的产出" → 绕过矛盾 = 丢弃最有价值的产出
- CLAUDE.md 原则1："定义不清楚时不写代码" → no-workaround 是原则1的反面表述

### 3. 边界条件
- 如果"矛盾"的定义被收窄（如仅限于定义级矛盾，不含实现层面的不一致），禁令的触发范围缩小
- 如果未来引入"可恢复的 workaround"概念（如显式标记 WORKAROUND + 自动过期机制），禁令的绝对性可能需要修正
- 当前禁令完全禁止 workaround，没有灰度

### 4. 下游推论
- no-workaround 的 runtime 强制仍缺乏（071号 #12），依赖 LLM 自律
- 与 testing-override.md 的配合：测试失败揭示定义冲突时，不修改实现让测试通过——这是 no-workaround 在测试域的投射

### 5. 谱系引用
- 005b号（对象否定对象语法规则）—— no-workaround 的概念根源
- 012号（谱系是发现引擎）—— "矛盾是最有价值的产出"
