# 核心禁令：严格性是蜂群的语法规则

此规则优先级与 `no-workaround.md` 相同。两者互补：
- no-workaround 禁止绕过矛盾（该停不停）
- 本规则禁止非严格的产出（该严格不严格）

## 范围

**所有蜂群产出**——代码、方案、审计、分析、对话、架构决策。无例外。
这不是一条工程建议，是蜂群的**语法规则**：非严格的产出在蜂群中是语法不合法的，
如同 005b 号禁止非对象否定。

## 何为"严格"

严格 = 概念清晰 + 声明与实际一致 + 没有未定义的模糊地带。

| 严格 | 非严格 |
|------|--------|
| 直面问题的本质形式 | "能修的修，不能修的承认" |
| 在约束条件下给出精确的存在论表达 | 用补丁/优化思路处理根本问题 |
| 完整解决，不留半成品 | "先修一半，后续再处理" |
| 代码能做什么就声明什么 | 声明代码不具备的能力 |
| 删除并用正确逻辑替换 | 在错误代码上加 workaround |
| 追问"严格的形式是什么" | 停留在"能不能做到" |

## 禁止的模式

1. **补丁思维**：在错误代码上加 workaround 让它"大致能工作"
2. **渐进式回避**："先保留旧逻辑，加个新分支处理边界情况"
3. **兼容性垫片**：保留已知错误的旧代码作为 fallback
4. **TODO 遗留**：修完后留下"后续优化"的尾巴
5. **声明膨胀**：在注释/文档中声明代码不具备的能力
6. **妥协方案**：面对严格性要求时退回到"能做到的先做"

## 要求的模式

1. **追问本质**：不是"能不能做到"而是"严格的形式是什么"
2. **重写**：如果逻辑是错的，删除并用正确逻辑替换
3. **完整**：一次修完，不留半成品
4. **诚实**：代码能做什么就声明什么，不多不少
5. **删除**：无用代码直接删除，不注释保留
6. **直面约束**：在约束条件下给出严格的表达，不因约束而降低严格性
