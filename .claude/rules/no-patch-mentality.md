# 核心禁令：严格方案，禁止补丁思维

此规则优先级与 `no-workaround.md` 相同。两者互补：
- no-workaround 禁止绕过矛盾
- 本规则禁止用补丁/优化思路处理任何问题

## 适用范围

所有工程动作（代码修改、hooks 修复、配置变更、架构调整）。无例外。

## 禁止的思维模式

- **补丁思维**：在错误代码上加 workaround 让它"大致能工作"
- **优化思维**："先保留旧逻辑，加个新分支处理边界情况"
- **兼容性垫片**：保留已知错误的旧代码作为 fallback
- **渐进式修复**："先修一半，后续再处理剩余"
- **TODO 遗留**：修完后留下"后续优化"的尾巴
- **声明膨胀**：在注释/文档中声明代码不具备的能力

## 要求的思维模式

- **重写**：如果逻辑是错的，删除并用正确逻辑替换
- **完整**：一次修完，不留半成品
- **诚实**：代码能做什么就声明什么，不多不少
- **删除**：无用代码直接删除，不注释保留

## 谱系依据

- 087号谱系：编排者 INTERRUPT 否定了 086号的保守全C方案（补丁思维的实例）
- 086号→087号的否定模式就是本规则的发生史
