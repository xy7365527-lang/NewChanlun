# 核心禁令：不允许向编排者提可自决的问题

此规则优先级与 `no-workaround.md` 相同。两者是一对：
- no-workaround 阻止绕过矛盾（该停不停）
- 本规则阻止不必要的上浮（该走不走）

## 四分法过滤（强制前置检查）

在向编排者提出**任何问题**之前，必须先执行四分法分类：

| 类型 | 定义 | 处理 | 允许提问？ |
|------|------|------|-----------|
| **定理** | 已结算原则的逻辑必然推论 | 自动结算 | ❌ 不允许 |
| **行动** | 不携带信息差的操作性事件 | 自动执行 | ❌ 不允许 |
| **选择** | 多种合理方案，需价值判断 | `/escalate` | ✅ 允许 |
| **语法记录** | 已在运作但未显式化的规则 | `/escalate` | ✅ 允许 |

## 强制输出格式（137号：替代自检序列）

否定性禁令对行为执行层无效（137号结算：RLHF基底约束）。以下不是"建议"，是**输出格式的硬约束**。

### 每次输出的结尾格式

每次完成一个 phase（commit、修复、分析、任何产出）后，输出**必须**以下列格式之一结尾：

**格式A：有后续行动时**
```
→ 接下来：[具体动作描述]
```
紧跟工具调用。`→ 接下来` 和工具调用之间不允许插入任何自然语言。

**格式B：所有可自主推进的工位已完成时**
```
[状态快照：N个工位完成，无生成态谱系，无待做行动]
```

**格式C：遇到真实矛盾时**
```
/escalate [矛盾描述]
```

**不存在第四种格式。** 任何不以 A/B/C 结尾的输出都是违规。
"总结后停下来"不是格式B——格式B需要明确声明"无待做行动"。

### 禁止的结尾模式

以下结尾模式是**语法不合法的**（不是"不建议"，是不合法）：

- 以问号结尾（除非在 `/escalate` 报告内）
- 以总结段落结尾但无行动声明
- 以"继续？"、"是否执行"、"待确认"等等待信号结尾
- 以选项列表结尾但未附带四分法分类

## 具体禁止的提问模式

- "是否现在执行 X？" — 如果 X 是可推进的独立工位且无阻塞项，直接做
- "更新 A 还是新建 B？" — 如果推导链可判断（同一链 = 更新，新链 = 新建），自己判断
- "是否需要 X？" — 如果 X 的必要性可从定义/谱系中推导，自己推导
- "先做 A 还是先做 B？" — 如果无数据依赖，并行做；有依赖，按依赖序做
- "这样可以吗？" — ceremony 已说明"开盘是自动的，不等待"
- "你指的是哪个？" — 如果编排者指出问题但未指定范围，而你能识别出多个相关缺口，全部修复。把范围确认反抛给编排者 = 把诊断工作外包给编排者。编排者的职责是指出方向，诊断和修复是蜂群的职责

## 允许的提问

- 定义之间的真实矛盾（走 `/escalate`）
- 需要编排者价值判断的方向选择（走 `/escalate`）
- 缺少外部数据/权限且无法自行获取
