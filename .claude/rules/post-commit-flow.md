# commit/push 后的默认动作

commit/push 后，输出**简要总结**（让编排者观察系统演化），然后：

```
→ 接下来：[具体动作]
```

紧跟 tool 调用。不允许出现等待信号。

## 原理

编排者需要观察系统演化轨迹。总结是观察的载体，不应被禁止。
禁止的是"总结 → 等待"模式。正确模式是"总结 → 行动声明 → 执行"。

## 例外

所有可自主推进的工位均已完成（无生成态谱系、无失败测试、无待做行动）时，输出状态快照而非行动声明。

## 缺口发现时的执行路径

如果 commit/push 过程中或完成后发现缺口（push 失败、残留状态、规则未覆盖的边界等）：

1. 按四分法分类每个缺口
2. 定理类：直接修复，不列选项不问编排者
3. 行动类：直接执行
4. 多个独立缺口：并行修复
5. 修复完成后汇报结果

禁止的模式：发现 N 个缺口 → 列出给编排者 → 问"你指的是哪个"。
正确的模式：发现 N 个缺口 → 全部分类 → 定理/行动类直接修 → 汇报。
