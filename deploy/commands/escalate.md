# /escalate — 矛盾上浮

当质询过程中发现不可弥合的**概念层**矛盾时，生成标准格式的矛盾报告呈交编排者。

## 用法

```
/escalate
```

在质询过程中发现概念层矛盾后调用。Lead根据当前上下文自动生成报告。

**调用前先判断矛盾层级：**
- 能否在不修改概念定义的前提下解决？→ 不要escalate，实现层自行解决。
- 需要补充工程规格但不改变概念含义？→ 不要escalate，lead自主执行小仪式。
- 需要修改概念定义本身？→ /escalate。

## 报告格式

```markdown
## 矛盾报告

### 矛盾描述
[什么跟什么冲突，为什么不可弥合]

### 双方论证
**立场A：** [agent X的结论及其定义依据]
**立场B：** [agent Y的结论及其定义依据（或质询中发现的不一致）]

### 涉及的定义
[列出相关定义条目，引用 `.chanlun/definitions/` 中的具体文件]

### 谱系比对结果
[是否有历史先例？如果有，上次怎么解决的，为什么这次不同]

### Lead的建议方案
[lead基于谱系比对和当前上下文，建议的分离方式。
编排者可以采纳、修改或否决。
如果lead也拿不准，明确说明"无建议，需要编排者从零判断"]

### 需要决断的问题
[精确地说明编排者需要判断什么——不是"这里有问题"，
而是"A和B哪个应该成立，或者是否需要概念分离"]
```

## 注意

- 「需要决断的问题」必须用领域语言（如缠论术语），不是代码语言
- lead应提供建议方案，降低编排者认知负荷——大多数情况编排者只需确认或微调
- 报告生成后自动保存到 `.chanlun/genealogy/` 目录（状态标记为生成态）
- 实现层和工程层矛盾不应该走到/escalate——如果发现上浮的矛盾其实是工程层的，lead直接自主处理
